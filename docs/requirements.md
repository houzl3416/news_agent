# NEWS GT 需求文档

## 1. 产品定位

**产品名称**: NEWS GT - AI 新闻真相认知引擎

**核心价值**: 利用自主AI智能体（Agent）集群，全自动重构信息传播和演变全貌，帮助用户看清真相的"完整拼图"。

**差异化**: 不仅告诉用户"对"或"错"，更重构信息传播和演变的全貌。

## 2. 核心功能需求

### 2.1 自主 Agent 调查集群

#### 监控哨兵 (Monitor Agent)
- **触发机制**:
  1. **用户提交（核心）**: 用户通过系统界面提交可疑新闻链接或事件描述，立即启动调查任务
  2. **周期性发现（演示）**: 对特选信源列表进行周期性巡查，发现关键词组合热度激增时自动触发
- **监控范围**: 主流媒体、特定行业论坛

#### 溯源猎手 (Source Hunter Agent)
- **职责**: 找到信息的"0号病人"
- **工具能力**:
  - 深度搜索
  - 社交媒体API集成
  - 互联网档案馆查询
  - 反向图像搜索
- **输出**: 信息的原始出处定位

#### 核查专家 (Verifier Agent)
- **职责**: 交叉验证信息真实性
- **数据源**:
  - 一级信源库（SEC监管文件、彭博/路透终端、官方新闻室）
- **核查方法**:
  - 数据提取
  - "沉默证据"检测（本该有但没有的证据）
- **输出**: 关键声明的核实结果

#### 叙事分析师 (Narrative Analyst Agent)
- **职责**: 分析信息如何被"加工"和"放大"
- **分析维度**:
  - 标题演变
  - 数字变化
  - 引语修改
  - 传播路径
- **检测能力**: 识别协同放大的可疑账户集群（水军网络）

#### 首席编辑 (Synthesizer Agent)
- **职责**: 汇总所有Agent发现，生成最终溯源报告
- **输出**: 格式化的调查报告

### 2.2 事件知识图谱 (EKG) - 核心壁垒

#### 数据结构
**节点类型**:
1. **事件 (Event)**:
   - 事件ID
   - 状态（发展中/已证伪/已证实）
   - 热度
   - 标签（金融/政治等）

2. **声明 (Claim)**:
   - 声明内容
   - 时间戳
   - 状态（待核实/已证实/已证伪）

3. **信源 (Source)**:
   - 名称
   - 类型（官媒/自媒体/匿名）
   - **信誉分 (Credit Score)** - 核心指标

4. **实体 (Entity)**:
   - 名称
   - 类型（人/组织/地点）

5. **物料 (Artifact)**:
   - URL
   - Tweet ID
   - 文件Hash

#### 关系类型
- HAS_CLAIM: 事件包含声明
- MADE_CLAIM: 信源发布声明
- MENTIONS: 声明提及实体
- IS_EVIDENCE_FOR: 物料是证据
- **REFUTES**: 声明证伪另一声明（核心关系）

#### 飞轮机制
- 每次调查结果沉淀为结构化知识
- 信源信誉分动态更新
- 实现毫秒级预警能力

#### MVP实现方案
- 技术选型: PostgreSQL + JSONB（务实方案）
- 核心表结构:
  - sources 表
  - claims 表
  - events 表
  - entities 表

### 2.3 真相即服务 (TaaS) - 生态壁垒

#### API能力
- 溯源调查API
- 实时传言风险评分API
- 信源信誉查询API
- 事实核查API

#### 目标客户
1. **金融领域**: 量化交易算法风险预警
2. **媒体领域**: 编辑室记者助手插件

## 3. 系统架构需求

### 3.1 技术栈
- **后端**: Python (Agent编排、业务逻辑)
- **数据库**: PostgreSQL (EKG存储)
- **API**: RESTful API (TaaS)
- **AI模型**: LLM集成（OpenAI/Claude等）
- **任务队列**: Celery/Redis（异步任务）

### 3.2 非功能性需求
- **性能**: EKG查询毫秒级响应
- **可扩展**: Agent系统可插拔设计
- **可靠性**: 调查任务持久化，支持断点恢复
- **安全性**: API密钥管理，数据加密

## 4. MVP范围界定

### 包含功能
1. ✅ 用户提交新闻链接触发调查
2. ✅ 5个Agent基础框架（接口定义）
3. ✅ EKG基础数据模型（PostgreSQL）
4. ✅ 基础TaaS API（至少2个端点）
5. ✅ 简单的调查报告生成

### 不包含功能
1. ❌ 完整的AI模型集成
2. ❌ 复杂的水军检测算法
3. ❌ 前端UI界面
4. ❌ 完整的监控哨兵周期性巡查
5. ❌ 生产级性能优化

## 5. 成功指标

### 技术指标
- 框架可运行，能处理一个完整的调查流程（端到端）
- EKG能正确存储和查询数据
- Agent间能正常通信和数据传递

### 演示指标
- 能演示：用户提交 -> Agent调查 -> 生成报告 -> EKG更新 -> 下次查询利用历史数据

## 6. 开发优先级

**P0（核心框架）**:
- Agent基类和编排器
- EKG数据模型
- 首席编辑Agent（报告生成）

**P1（MVP闭环）**:
- 溯源猎手Agent（基础搜索）
- 核查专家Agent（基础验证）
- TaaS API基础端点

**P2（演示增强）**:
- 监控哨兵Agent
- 叙事分析师Agent
- 信誉分更新机制
